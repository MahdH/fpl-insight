Introduction
A lot of people play Fantasy Premier League, and they need good data to win. The official FPL website gives a lot of numbers, but it is hard for normal users to combine all that data to see who is actually playing well. The main goal of this project was to build a tool that does the math for them. Instead of just looking at who scored a goal last week, our app looks at how dangerous a player is and how hard their next game is. This report will talk about the tools we used to build it, how our custom math works, the steps we took to put it on the internet, and the problems we had to fix along the way.
Development Stack
For this project, we used Artificial Intelligence to help with the planning and writing process. I used Gemini to plan the project steps and to learn about industry best practices for system design. For writing the actual code, we used Google Antigravity as our main development environment. To make the website look good and design the user interface, we used Google Stitch MCP to create our UI style guides. Using these tools helped speed up the work and made sure the code followed professional standards.
System Architecture and Technologies
The very first step I took was creating an isolated workspace on my computer, which is a standard industry practice. Instead of installing tools globally where they might conflict with other projects, I created a Python virtual environment. You can think of a virtual environment as a soundproof room where whatever happens inside doesn't affect the rest of the house. Inside this isolated room, I installed the core dependencies needed to run a web server.

Algorithms and Logic
I wrote my own math formulas to find the best players. The first one is the Dynamic Striker Index. This algorithm looks at a forward's recent form and total points, but it also changes their score based on where they are playing next. If their team attacks really well at home, they get bonus points. if you want to know if a striker is a good pick, you have to look at their points, then manually check if their next game is home or away, and then try to remember if their team is actually good at attacking in that stadium. Our algorithm fixes this by calculating it automatically. It looks at a forward's recent form and total points, but it also changes their score based on where they are playing next. If their team attacks really well at home, they get bonus points.
Our second formula is the Performer Index. This one ignores the player's price. It only looks at how well they are playing right now based on their "per 90 minutes" stats. This makes sure we don't pick players who just got lucky in one game. In the standard app, advanced stats like ICT (Influence, Creativity, Threat) and BPS (Bonus Points System) are hidden deep inside player profiles. Also, the official app only shows total stats, so users easily get tricked into buying players who just played a lot of games or got lucky once. Our algorithm fixes this by ignoring the player's price and total points.
To suggest transfers, I built a Set Difference engine. When a user types their ID, I get their 15 players and put them in a Python set. Then I loop through all 800 players in the game. If a player is not in the user's set, I run them through the Performer Index. Normally, making a transfer in the FPL app is very slow. You have to look at your team, go to the transfer market, and manually click on different players to compare them, making sure you don't already own them. Our engine does this instantly. At the top of the Transfer Lab page, I placed the most important feature, which is the Top Striker recommendation. I made this a large hero card with the player's photo in the background to grab the user's attention. Instead of just showing the player's name, I put my custom algorithm score right next to the standard projected points and ownership percentage. I did this so users can easily compare my smart data with normal FPL data at a single glance. Right below the main player, I added smaller rows for the second and third best options, so users have backup choices without needing to scroll down too much.
In the middle of the screen, I created two smaller action cards for quick decisions. One card is for suggesting players to transfer out, and the other is for targeting good upcoming matches. I used color psychology here to help users understand the data faster without reading a lot of text. For example, if a player is injured and needs to be sold, the transfer out card uses red icons and red text. For the target fixture card, I used a visual bar with green dots to show the difficulty level, which makes it much faster to understand how easy or hard a match will be.
Finally, at the bottom of the page, I added a clean list for the Top Performers and a main navigation bar. The Top Performers section uses a simple ranked list because it saves screen space while still showing the live index scores for the best players in the league. The bottom navigation bar includes a glowing green icon for the Transfer Lab to show the user exactly which page they are currently looking at, along with a simple button to switch over to the My Team pitch view. All of these design choices work together to make complicated math and data look simple, clean, and professional.
